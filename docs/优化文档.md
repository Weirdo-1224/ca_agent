
基于对后端系统的全面审查，结合比赛评分标准（多 Agent 协作 35% + 技术深度 25% + 业务价值 20%），以下是按优先级排序的关键优化点分析：

---

## 后端系统优化分析报告

### P0 - 最高优先级（直接影响 35% 评分：多Agent协作与输出可信度）

#### 1. 增强 Agent 可观测性：Prompt / Token 追踪

**现状问题**：`AgentRunTracer` 只记录了执行时间和状态，**缺少 Prompt 内容、Token 消耗、LLM 原始响应**的追踪。评分标准明确要求"每个 Agent 的 Prompt、输入、输出、决策过程、Token 消耗均有日志/Trace 可查"。

**建议方案**：
```
agent_run 表增加字段：
- prompt_tokens INT
- completion_tokens INT  
- total_tokens INT
- system_prompt TEXT
- user_prompt_preview TEXT (前500字)
- raw_response_preview TEXT (前500字)
```
在 `SpringAiModelChatGateway.call()` 中记录 token 使用量，`AgentRunTrace` 增加相应字段。

---

#### 2. 完善异步执行的失败状态管理

**现状问题**：[WorkflowService.runAsync()](file:///d:/Java_Project/CA_Agent/src/main/java/org/example/ca_agent/service/WorkflowService.java) 中工作流异常被 catch 后只记录日志，**任务状态永远停留在运行中状态**，前端轮询永远不会停止。

```java
// 当前代码 - 异常被吞没
} catch (Exception e) {
    log.error("[Async] Workflow failed for task: {}", taskInput.getTaskId(), e);
    // 缺少：将任务状态更新为 FAILED
}
```

**建议方案**：catch 块中更新任务状态为 `FAILED` 并保存错误信息到数据库。

---

#### 3. LLM 调用缺少超时和重试（非解析重试）

**现状问题**：[StructuredLlmService](file:///d:/Java_Project/CA_Agent/src/main/java/org/example/ca_agent/service/StructuredLlmService.java) 只对 JSON 解析失败做了重试，**LLM 网络调用本身没有超时控制和重试**。大模型调用可能hang住导致整个工作流阻塞。

**建议方案**：
- `SpringAiModelChatGateway` 增加调用超时（60s）
- 对 HTTP 5xx / 超时错误增加指数退避重试（最多 2 次）

---

### P1 - 高优先级（影响 25% 评分：技术深度与工程完整度）

#### 4. StateAssembler 全量删除重建 → 增量更新

**现状问题**：[saveState()](file:///d:/Java_Project/CA_Agent/src/main/java/org/example/ca_agent/assembler/StateAssembler.java) 每次保存时**删除 6 张表的所有记录再全部重新插入**，每个 Agent 执行后都触发一次。如果有 10 条 evidence + 10 条 claim + 6 条 agent_run，每次保存要做 ~30 次 DELETE + ~30 次 INSERT。

**影响**：性能差、数据库压力大、存在并发写入风险。

**建议方案**：改为增量保存 — agent_run 只追加新记录，evidence/claim 等只在对应 Agent 执行后才保存。

---

#### 5. 上下文管理：ReviewerAgent 接收完整状态 JSON

**现状问题**：[ReviewerAgent](file:///d:/Java_Project/CA_Agent/src/main/java/org/example/ca_agent/agent/ReviewerAgent.java) 将 `JsonUtils.toJson(state)` 整个状态序列化后作为 Prompt 发送给 LLM。当证据池较大时，这个 JSON 可能超出模型的 context window（当前 max-tokens=2048 只限输出，但输入也有限制）。

**建议方案**：
- 对 Reviewer 的输入做上下文压缩/摘要
- 只传递报告章节 + 证据索引 + claim 列表，不传原始 rawText
- 增加 token 计数预检，超限时自动截断

---

#### 6. 幻觉控制增强：自一致性校验

**现状问题**：评分标准提到"自一致性校验、引用强制、超长上下文分片"。当前 `AgentOutputValidator` 只验证 evidenceIds 存在性，**未验证 evidenceId 关联内容是否与 claim 语义相关**。

**建议方案**：
- 验证 claim 文本与 evidence contentSnippet 的语义关联性（可用简单关键词匹配）
- 报告章节 content 中引用的 evidenceId 数量检查（空引用告警）

---

### P2 - 中优先级（影响业务价值和稳定性）

#### 7. 搜索采集并发优化

**现状问题**：[CollectorAgent](file:///d:/Java_Project/CA_Agent/src/main/java/org/example/ca_agent/agent/CollectorAgent.java) 对多个产品和多个查询是**顺序执行**的，网页读取也是串行。真实模式下 2 个产品 × 5 个查询 = 10 次搜索 + 10 次网页读取，串行耗时巨大。

**建议方案**：使用 `CompletableFuture` 并行执行多产品的搜索和网页读取，设置总体超时。

---

#### 8. 请求参数校验

**现状问题**：[AnalysisTaskController](file:///d:/Java_Project/CA_Agent/src/main/java/org/example/ca_agent/controller/AnalysisTaskController.java) 的 `@RequestBody TaskCreateRequest` 没有 `@Valid` 注解，`TaskCreateRequest` 也没有 `@NotBlank` / `@Size` 等校验注解。恶意或错误输入会导致下游 NPE。

**建议方案**：增加 JSR-303 校验注解 + `@Valid`。

---

#### 9. GlobalExceptionHandler 不记录堆栈

**现状问题**：[GlobalExceptionHandler](file:///d:/Java_Project/CA_Agent/src/main/java/org/example/ca_agent/common/GlobalExceptionHandler.java) 对非业务异常返回 "Internal server error"，但**不记录异常堆栈**，线上问题无法排查。

```java
@ExceptionHandler(Exception.class)
public Result<Void> handleException(Exception exception) {
    return Result.fail("Internal server error"); // 缺少 log.error
}
```

---

#### 10. 外部服务熔断保护

**现状问题**：`MetasoSearchClient` 和 LLM 调用都没有熔断机制。如果秘塔 API 持续 5xx，系统会反复请求导致线程阻塞。

**建议方案**：增加简单的熔断器（连续 N 次失败后短路，自动降级到 mock 数据）。
